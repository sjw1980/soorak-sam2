## 1. 목표 및 적용 범위

- 목표: 소스코드·빌드 아티팩트·문서 등 형상 항목(Configuration Items, CI)의 무결성, 추적성, 재현성, 변경 통제(변경요청→승인→이행) 및 베이스라인 관리를 보장한다.
- 적용 범위: CLI Calculator 프로젝트의 소스코드, 빌드 스크립트, CI 정의, 테스트 아티팩트, 릴리스 번들, 관련 문서(설계·요구·릴리스 노트).

참조: [docs/MAN-3/MAN3-PP-0001-project-plan.md](docs/MAN-3/MAN3-PP-0001-project-plan.md), [docs/PROJECT-KICKOFF.md](docs/PROJECT-KICKOFF.md)

## 3. 형상 항목(CI) 분류 및 베이스라인

- CI 분류:

  - 소스코드: src/ 및 테스트코드
  - 빌드 정의: ci/, Dockerfile, GitHub Actions 워크플로
  - 아티팩트: 컨테이너 이미지, 릴리스 번들
  - 문서: 요구사항·설계·릴리스 노트·추적성 매핑표

- 베이스라인 유형:
  - 요구사항 베이스라인
  - 설계(아키텍처/상세) 베이스라인
  - 구현/단위검증 베이스라인
  - 통합/릴리스 베이스라인(태그/릴리스에 의해 관리)

베이스라인은 Git 태그와 릴리스(예: v1.0.0), 빌드 아티팩트(컨테이너 레지스트리의 immutable tag 또는 digest)로 확보한다.

## 4. 변경 통제 절차 (CR workflow)

1. 변경 식별: 변경 요청(CR)은 GitHub Issue 템플릿(CR)으로 등록한다. 최소 항목: 요약, 변경사유, 영향받는 CI 목록, 제안된 해결방안, 제출자.
2. 초기 분류: Configuration Manager가 우선 분류(긴급/정규)하고 담당자 지정.
3. 영향분석: Requirement Owner / Developer / QA가 영향 분석 수행(요구사항·테스트·릴리스 영향 포함).
4. 승인결정: Project Lead(또는 승인자 유저)가 승인/반려/보류 결정.
5. 이행: 승인된 CR은 feature/ 또는 bugfix/ 브랜치에서 구현, PR 생성(검토자 최소 1인, CI 통과 필요).
6. 병합 및 베이스라인 갱신: PR Merge 후 필요 시 새로운 베이스라인 태그 생성 및 릴리스 등록.
7. 기록 유지: 모든 CR 및 관련 증거(코드, PR, CI 로그, 테스트 결과)는 변경 통제 기록(WP-13-21) 및 추적성 기록(WP-13-22)에 보관.

필수 증거: 영향분석 문서, PR 리뷰 기록, CI 빌드/테스트 결과, 변경 통제 기록

## 5. 도구 및 저장 위치

- 버전 관리: Git (GitHub) — 중앙 저장소
- CI/CD: GitHub Actions (빌드, 테스트, 릴리스 업로드)
- 컨테이너 레지스트리: GitHub Container Registry (GHCR) 또는 지정 레지스트리
- 아티팩트 저장: GitHub Releases, 빌드 아티팩트 스토리지
- 이슈/CR: GitHub Issues / Pull Requests

접근 제어: 주요 브랜치(main, release/\*)에 대한 브랜치 보호 규칙 적용(Required reviews, CI pass 등). 권한 관리: GitHub 조직/팀 권한 사용.

## 6. 저장·백업·전달 체계

- 저장 위치 요약:

  - 소스코드/이슈: GitHub repo
  - 빌드 아티팩트: GitHub Releases / GHCR
  - 문서: repo 내 docs/ 및 릴리스 패키지

- 백업 정책(초안):
  - 소스코드 미러/백업: 일별(자동) 미러를 외부 오브젝트 스토리지(예: offsite cloud storage) 또는 내부 NAS에 보관. 보관기간 90일(기본).
  - 릴리스 아티팩트: 릴리스 시점에 아카이브 저장(보관기간 1년 이상 권장).
  - CI 로그/테스트 결과: 90일 보관.

전달 절차: 외부 전달물(고객 인도)은 릴리스 패키지와 릴리스 노트로 구성, 전달 매체는 컨테이너 이미지 또는 압축된 릴리스 번들.

## 7. 형상 감사 및 심사 기준

- 형상 감사 주기: 주요 게이트(요구사항 베이스라인·설계 베이스라인·릴리스 게이트)에서 수행.
- 심사 항목(예): 베이스라인 완전성(관련 CI 포함 여부), 추적성(요구→설계→테스트), 변경 통제 기록 존재 여부, 빌드·테스트 증거.

## 8. 측정지표(KPI)

- CR 처리시간(평균)
- 베이스라인 생성 대비 증거(%)
- CI 빌드 성공률(%)
- 릴리스별 이슈 재개수(회귀 지표)
- 백업 성공률 및 복구 검증 주기

측정 주기: 주간/월간 보고. KPI 보고는 Project Lead 및 QA에 제출.

## 9. 교육 및 운영

- 형상관리 절차 및 브랜치 규칙, CR 작성 템플릿 교육을 개발팀과 QA 팀에 시행한다(킥오프 이후 1주 내).
