# WP-08-27 문제 관리 계획서

- 버전: v1.0 / 2026-04-19
- 상태: Draft
- ASPICE 참조: SUP-9 BP1

## 1. 목적

이 문서는 프로젝트의 문제(Problem) 보고·분류·우선순위화·해결·종료까지의 관리 정책과 절차를 정의한다. 특히 문제 상태 모델, 경보(알림) 통지 전략, 긴급 해결(Hotfix/롤백) 전략을 명확히 하여 신속하고 일관된 문제 처리를 보장한다.

## 2. 범위

이 계획서는 본 소프트웨어 제품의 개발·통합·테스트·배포·운영 단계에서 식별되는 모든 문제(설계·코드·빌드·릴리스·운영 결함 포함)에 적용된다. 외주·서드파티 제품의 문제는 해당 공급자 대응절차를 병행 적용한다.

## 3. 참조문서

- WP-13-07 문제 기록 템플릿
- WP-13-19 검토 기록
- SUP-9 프로세스 규정서

## 4. 정의

- 문제(Problem): 재현 가능한 결함, 비정상 동작, 안전·보안 위협, 심각한 성능 저하 등으로 프로젝트 목표·릴리스에 영향을 주는 사항.
- 보고자(Reporter), 문제 소유자(Owner), Triage 팀, CCB(Change Control Board) 등 역할은 본 문서의 역할 정의를 따른다.

## 5. 문제 상태 모델 (Problem Status Model)

본 섹션은 문제의 상태(state)와 각 상태에서의 책임·행동·전이(transition)를 정의한다.

상태 목록 및 설명:

- New (신규): 문제 보고가 생성된 초기 상태. Reporter가 문제를 등록하면 자동 할당.
- Acknowledged (접수): Triage 팀이 문제 접수를 확인하고 우선순위·심각도 초안 지정.
- Triaged / Assigned (분류·할당): 근본 원인 조사 담당(Owner)과 우선순위가 확정되어 담당자에게 할당됨.
- In Progress (조치중): 담당자가 해결 조치(수정, 회피책 적용 등)를 수행 중.
- Escalated (에스컬레이션): 심각도/영향으로 인해 상위 조직(엔지니어링 리드·프로젝트 매니저 등)으로 이관된 상태.
- Suspended (보류): 외부 의존성(서드파티·고객·환경)으로 인해 조치가 중단된 상태. 재개 조건과 책임자를 기록해야 함.
- Resolved (해결됨): 수정 또는 회피책 적용으로 문제를 해결했으며, 해결 내용(커밋/패치/설정 변경 등)을 기술.
- Verified (검증됨): QA/검증 담당자가 해결을 재현 불가·검증 통과로 확인.
- Closed (종료): 문제의 종료 기준(검증 통과 및 릴리스 반영 완료) 충족 시 공식 종료.
- Reopened (재오픈): 검증 후 동일 증상 또는 관련 증상 재발 시 문제를 재오픈.

전이 예시:

- New → Acknowledged → Triaged/Assigned → In Progress → Resolved → Verified → Closed
- In Progress → Escalated → (Assigned →) In Progress
- In Progress → Suspended → In Progress
- Verified → Reopened → In Progress

상태별 책임 요약:

- Reporter: 문제 등록(로그·스크린샷·재현절차 포함).
- Triage 팀: 접수·심각도 초기 지정·간단 재현 시도.
- Owner(문제 소유자): 조치 계획 수립·수정 실행·상태 갱신.
- QA/검증: 해결 검증 및 검증 결과 기록.
- CCB/릴리스 매니저: 긴급 수정(Hotfix) 승인·릴리스 반영 결정.

기록 요건(모든 상태 변경 시): 문제 ID, 상태 변경자, 시간, 간단 사유/조치 내역, 관련 커밋/빌드/릴리스 참조.

## 6. 경보 통지 전략 (Alert Notification Strategy)

목표: 심각도에 따라 관련 이해관계자에게 적절한 채널·시간내(acknowledgement) 통지를 수행하고, 미응답 시 자동·수동 에스컬레이션을 실행한다.

1. 심각도(Severity) 정의 및 예시

- Critical (치명): 시스템 중단, 데이터 손실, 안전 영향 — 즉시 대응 필요.
- High (높음): 주요 기능 마비 또는 주요 사용 시나리오 차단 — 빠른 대응 필요.
- Medium (중간): 주요 영향은 있으나 우회/임시조치 가능.
- Low (낮음): 경미한 기능·문서화성 이슈.

2. 통지 채널

- 이슈 트래커(JIRA/GitHub Issues): 모든 문제의 단일 기록소(최우선).
- 이메일: 공식 통지 및 이력 저장용.
- 실시간 메신저(Teams/Slack): 빠른 알림·긴급 채널.
- SMS/전화(온콜·핵심 담당자): Critical 시 우선 호출.

3. 통지 규칙 및 SLA

- Critical:

  - 즉시(0분) 자동 알림: 온콜 엔지니어, 엔지니어링 리드, 프로젝트 매니저에게 SMS/전화 및 메신저 알림.
  - 응답(ack) 시간: 15분 이내. 미응답 시 15분 후 엔지니어링 책임자, 30분 후 경영/릴리스 책임자에게 에스컬레이션.
  - 상태 업데이트: 최초 1시간 이내 상황공유, 이후 시간 단위로 상태보고(핵심 이해관계자).

- High:

  - 자동 알림: 담당팀 메신저 채널 + 이메일.
  - 응답(ack): 60분 이내. 미응답 시 4시간 내 매니저에게 에스컬레이션.

- Medium:

  - 이메일 및 이슈 트래커로 통지.
  - 응답: 업무일 기준 1영업일 이내.

- Low:
  - 주간 요약 또는 다음 스프린트로 이관.

4. 에스컬레이션 절차

- 미응답·중요 임계치 도달 시 자동화 스크립트 또는 온콜 담당자가 다음 레벨로 수동 에스컬레이션 수행.
- 에스컬레이션 시 반드시 이슈에 에스컬레이션 사유·시간·에스컬레이트 대상 기록.

5. 통지 내용(템플릿 요약)

- 제목: [문제ID] [심각도] 한 줄 요약
- 본문: 문제 요약, 영향 범위, 재현 단계(요약), 현재 상태, 담당자, 링크(이슈 트래커), 긴급 연락처

## 7. 긴급 해결 전략 (Emergency Resolution Strategy)

긴급(주로 `Critical`) 문제 발생 시 신속한 격리·임시복구·영구해결을 목표로 한다. 아래 절차를 따르되 안전·보안 관련 문제는 별도의 안전 절차를 우선 적용한다.

1. 즉시 조치(Containment)

- 영향 범위 파악(서비스·고객·데이터 영향).
- 추가 피해 방지: 서비스 일시 중단·트래픽 차단·기능 비활성화 등 최소 권한 조치.
- 임시 대책(Workaround) 문서화 및 배포.

2. 임시 릴리스/핫픽스(Hotfix) 흐름

- 긴급 브랜치 생성: `hotfix/<issueID>` 형태.
- 수정 개발·빠른 단위/통합 테스트(자동화 가능한 최소 테스트 실행).
- 긴급 CCB 회의: 긴급 패치 승인(원격 컨퍼런스 가능) — CCB 구성: 프로젝트 매니저, 엔지니어링 리드, 릴리스 매니저, QA 리드.
- 검증 환경(스테이징)에서의 검증 후 프로덕션 적용.

3. 롤백(Backout) 계획

- 모든 긴급 배포 전에는 롤백 절차와 백업(데이터·설정) 확보.
- 롤백 결정 기준(예: 롤아웃 후 30분 내 주요 서비스 이상 지속 시)은 미리 합의.

4. 커뮤니케이션

- Critical 사건은 시간 단위로 상태 업데이트 제공(이슈 코멘트 + 메신저 핀업 + 이메일).
- 사건 종료 후 24시간 이내 초기 사고 보고(임시 보고서) 제출.

5. 사후 조치(Post-Incident)

- Root Cause Analysis(RCA) 수행: 책임자 지정, RCA 보고서 작성(5영업일 내 초안).
- 장기 대응: 영구 패치, 테스트 보강, 릴리스 계획 반영.
- 예방대책 문서화 및 관련 절차·체크리스트 갱신.

6. 안전·규제 이슈 우선 처리

- 안전·보안 관련 문제는 별도 안전 매니저 및 규제 대응 팀과 즉시 연계하고, 법적·규제 보고 요건을 충족하도록 대응한다.

## 8. 역할 및 책임

- Reporter: 문제 등록·초기 정보 제공.
- Triage/On-call 팀: 접수·초기 분류·긴급 판단.
- Owner(문제 담당자): 해결 계획 수립·조치 실행·상태 갱신.
- QA: 해결 검증·재현 시나리오 확인.
- 릴리스 매니저/CCB: Hotfix 승인·릴리스/롤백 결정.
- 프로젝트 매니저: 주요 이해관계자 커뮤니케이션·에스컬레이션 관리.

## 9. 측정지표 및 보고

- 주요 KPI:
  - MTTA (Mean Time To Acknowledge): 평균 최초 응답 시간.
  - MTTR (Mean Time To Resolve): 평균 해결 소요 시간.
  - 재발률: 동일 이슈 재오픈 비율.
- 보고 주기: Critical 사건은 실시간/시간단위; 주간 문제 요약 리포트는 매주 발행.
