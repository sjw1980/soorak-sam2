## 1. 복구 대상 식별

복구 대상(우선순위):

1. 소스코드 및 Git 저장소 메타데이터(이슈·PR 포함)
2. 빌드 정의 및 CI 워크플로
3. 빌드 아티팩트(컨테이너 이미지, 릴리스 번들)
4. 테스트 결과 및 로그
5. 설계·요구·릴리스 문서 및 추적성 매핑표

## 2. 복구 우선순위 및 심각도 정의

- Critical: 중앙 저장소 손상 또는 릴리스 불가(서비스 중단 수준)
- Major: 특정 아티팩트(컨테이너 이미지 등) 손실로 기능 배포 지연
- Minor: 개별 문서 손실 등

## 3. 복구 절차(요약)

1. 사고 인지 및 초기통지: Configuration Manager가 사고를 접수·분류하고 Project Lead와 주요 담당자에게 통지한다.
2. 초기 복구 결정(긴급/정규): 사고 심각도에 따라 복구 우선순위 결정.
3. 복구 소스 식별: 최신 백업(또는 릴리스 아카이브, 로컬 복제본)을 식별.
4. 복원 수행: 대상 CI에 대해 복원 절차 실행(아래 세부 참조).
5. 복구 검증: 자동 테스트·수동 검증으로 무결성·기능성 확인.
6. 기록 및 보고: 복구 활동 로그·원인분석(Root Cause Analysis), 재발방지 조치 등록.

### 3.1 소스코드(Git) 복원

- 사본(로컬 클론 또는 미러)이 존재하면 해당 사본을 임시 저장소에 푸시하여 중앙을 복구한다.
- 중앙 복제본이 없고 GitHub 서비스 문제일 경우 GitHub 지원 채널을 통해 복구를 요청하되, 로컬 백업으로 우회 복원 수행.
- 릴리스 태그/릴리스 아카이브는 GitHub Releases에서 재다운로드 가능.

### 3.2 컨테이너 이미지/아티팩트 복원

- GHCR(또는 레지스트리)에서 이미지 pull; 만약 레지스트리 손상 시 오프사이트 아카이브로부터 재푸시.
- 릴리스 번들은 GitHub Releases 또는 별도 아카이브 저장소에서 복원.

### 3.3 CI/CD 및 빌드 환경 복원

- CI 워크플로(코드로 정의된 pipeline)는 저장소 복원 후 자동으로 복구.
- 빌드 환경(종속성 등)은 요구사항에 따른 재구성 및 빌드 스크립트로 재생성.

## 4. 복구 일정 목표 (초안)

| 심각도   | RTO (목표 복구시간) | RPO (허용 데이터 손실) |
| -------- | ------------------: | ---------------------: |
| Critical |             8 hours |               24 hours |
| Major    |            24 hours |               24 hours |
| Minor    |            72 hours |                 7 days |

위 수치는 프로젝트 상황(리소스)에 따라 조정 가능하다.

## 5. 중대한 의존성

- GitHub(저장소/이슈/릴리스)
- 컨테이너 레지스트리(GHCR 또는 지정 레지스트리)
- CI 서비스(GitHub Actions)
- 외부 오브젝트 스토리지(오프사이트 백업)

## 7. 백업 항목 및 보관 정책(초안)

| 백업 항목           | 백업 주기           | 보관 기간 | 저장 위치                         |

## 8. 검증 및 복구 연습

- 정기 복구 연습: 분기별(권장) 또는 최소 연 1회 실제 복원 시나리오 실행.
- 검증 방법: 복원 후 자동 테스트(통합 테스트·기본 시나리오) 실행 및 수동 확인.

## 9. 특수 장비/자원

- 오프사이트 백업을 위한 클라우드 스토리지 계정, 대체 빌드 서버(필요시), 접근 권한을 가진 관리자 계정.

## 10. 비용 추정(초안)

- 소규모 프로젝트 기준: 백업 스토리지·레지스트리 보관 비용이 주된 항목이며, 연간 예산은 사용량에 따라 책정. 세부 견적은 운영팀과 협의 후 확정.

## 11. 기록 및 보고

- 복구 완료 후에는 복구 보고서(원인, 영향, 소요시간, 재발방지)를 작성하여 Project Lead와 QA에 제출.
