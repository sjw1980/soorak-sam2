## 2. 수명주기 모델/방법론

### 권장 모델: Iterative V-Model (증분형 V-모델)

프로젝트는 ASPICE MAN-3 요구사항을 만족하면서 리스크를 조기에 완화할 수 있도록 '증분형 V-모델(Iterative V-Model)'을 채택합니다. 이 모델은 전형적인 V-모델의 검증·검증(verification & validation) 활동을 반복적인 증분(Iteration)으로 수행하여 각 증분마다 요구사항 → 설계 → 구현 → 단위검증 → 통합검증을 수행합니다.

이유:

- ASPICE 요구사항(특히 MAN-3)은 명확한 게이트(설계검토·검증증거)를 요구하므로 V-모델의 검증/검증 구조가 적합합니다.
- 증분 접근은 주요 리스크(요구사항 불확실성·통합 문제)를 조기에 발견·완화합니다.

### 모델 개요 및 페이즈

각 증분(Iteration)은 아래의 페이즈를 포함합니다. 프로젝트 전체는 여러 증분으로 구성되며, 주요 마일스톤은 [5. 마일스톤]에 정의된 게이트와 연동됩니다.

- Phase 0: Initiation / Kickoff

  - 진입조건: 프로젝트 헌장·자원 배정
  - 종료조건: 킥오프 회의 완료, 초기 플랜 승인
  - 산출물: Kickoff 문서, 프로젝트 계획서, 리스크 로그

- Phase 1: Requirements Definition & Allocation

  - 진입조건: 킥오프 완료
  - 종료조건: SRS(요구사항 명세) 베이스라인 승인
  - 산출물: 요구사항 목록(SRS), 요구사항 ID, 요구사항-테스트 추적성 매트릭스, 요구사항 리뷰 기록

- Phase 2: Architecture & Detailed Design

  - 진입조건: 요구사항 베이스라인
  - 종료조건: 상세설계(하부단위 포함) 승인
  - 산출물: 아키텍처 문서, 인터페이스 명세서(WP-17-08 참조), 상세 설계서, 설계 리뷰 기록

- Phase 3: Implementation / Unit Development

  - 진입조건: 설계 베이스라인
  - 종료조건: 단위검증(유닛 테스트/정적분석) 합격, 코드 리뷰 완료
  - 산출물: 소스코드, 단위테스트 코드 및 실행결과, 정적분석 리포트, 유닛 검증 기록

- Phase 4: Integration & Verification

  - 진입조건: 단위검증 완료(필요 유닛)
  - 종료조건: 통합검증(인터페이스·시나리오 기준) 합격
  - 산출물: 통합 빌드, 통합 테스트 케이스·결과, 통합 검토 기록

- Phase 5: System Verification & Validation (Acceptance)

  - 진입조건: 통합검증 완료
  - 종료조건: 수용기준 충족(릴리스 후보 선정)
  - 산출물: 시스템 테스트 결과, 적격성시험(QTC) 결과, 수용 테스트 기록

- Phase 6: Release & Maintenance
  - 진입조건: 수용 승인
  - 종료조건: 정식 릴리스 배포
  - 산출물: 릴리스 패키지, 릴리스 노트, 베이스라인 등록, 유지보수 계획

### 게이트(검토 지점) 및 필수 증거

각 주요 게이트(예: 요구사항 베이스라인 승인, 설계리뷰, 코드완료, 통합완료, 릴리스 승인)에서는 아래 증거를 제출해야 합니다.

- 요구사항 게이트: SRS, 요구사항-테스트 추적성 매트릭스, 요구사항 리뷰 기록
- 설계 게이트: 아키텍처·상세설계 문서, 인터페이스 명세, 설계 리뷰 회의록
- 코드/단위검증 게이트: 소스코드, 단위테스트 결과, 정적분석 리포트, 코드리뷰 기록, 커버리지 리포트
- 통합 게이트: 통합 빌드, 통합 테스트 사례·결과, 통합 리뷰 기록
- 릴리스 게이트: RC 빌드, 시스템 테스트·적격성 결과, 보안·배포 지침, 릴리스 노트

모든 게이트 기록은 검토 기록(WP-13-19), 변경 통제 기록(WP-13-21) 및 추적성 기록(WP-13-22)에 보관합니다.

### 역할 및 책임(요약)

- Requirement Owner: 요구사항 수집·확정 책임
- Lead Architect: 아키텍처·설계 승인 주체

- Configuration Manager: 베이스라인·릴리스 관리

### ASPICE(MAN-3 BP2) 정렬 근거

본 수명주기 모델은 MAN-3 BP2 요구사항(프로젝트 수명주기 모델의 선택·정의·설명)에 부합합니다. 특히:

- 단계(phase)와 산출물(artifacts)을 명시하여 각 게이트에서 증거를 제시할 수 있도록 설계했습니다.
- 역할·검토 책임 및 검토 증거(레코드)를 정의하여 ASPICE 심사 시 요구되는 프로세스 증거(Review records, traceability)를 확보합니다.
- 반복적 증분 방식을 통해 리스크 관리를 용이하게 하고, 주요 게이트에서의 엄격한 검증으로 품질 확보를 병행합니다.

### 실행 권고사항(간단)

- 증분 주기: 1~2주 권장(프로젝트 규모에 따라 조정)
- 각 게이트는 체크리스트(templates/wp-13-19.md 등)를 사용하여 표준화
- 추적성: 요구사항→설계→테스트 케이스→테스트 결과의 양방향 매핑을 지속 갱신

(참고: 필요 시 세부 템플릿·체크리스트는 `wp-templates/` 폴더의 관련 문서를 사용하여 표준화합니다.)

## 11. WBS 요약

프로젝트의 상세한 업무 분해 구조(WBS)는 [WP-14-09](WP-14-09.md)에서 정의되어 있습니다. 주요 작업 그룹은 아래와 같습니다.

- **1.2 Requirements**: 요구사항 수집·검토·베이스라인 (SRS)
- **1.3 Design**: 아키텍처 및 상세 설계 (인터페이스 명세 포함)
- **1.4 Implementation**: 단위 개발 및 정적/동적 검증
- **1.5 Integration & Verification**: 통합 빌드·통합 테스트·시스템 검증
- **1.6 Release**: RC 빌드·릴리스 패키징·배포
- **1.7 Support Activities**: 형상관리(SUP-8), QA·검토(SUP-1), 리스크 관리(SUP-9)

각 작업 패키지는 WP-14-09에 일괄적으로 매핑되어 있으며, 산출물(예: WP-04-04 아키텍처, WP-04-05 상세설계, WP-17-11 요구사항 명세 등)과 연결되어 있습니다. 상세 산출물 매핑은 WP-14-09의 "산출물 매핑" 표를 참조하십시오.
