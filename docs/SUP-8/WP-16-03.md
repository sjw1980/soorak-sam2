# WP-16-03: 형상 관리 시스템 구축 기록

## 1. 목적

이 문서는 soorak-sam2 프로젝트에 적용된 형상 관리(Configuration Management, CM) 시스템의 구축 기록과 운영 지침을 문서화한다. 주로 저장소 구성, 접근 권한 설정, 변경 이력 관리 방법(베이스라인/태깅/이력조회)을 정의하여 형상 무결성, 추적성 및 감사가능성을 확보하는 것을 목적으로 한다.

## 2. 범위

- 적용 대상: 소스 코드, 설계·요구 문서, 테스트 산출물, 빌드 스크립트, 릴리스 아티팩트
- 제외 대상: 외부 상용 바이너리(서드파티 패키지는 별도 저장소/아카이브로 관리)

## 3. 용어 및 약어

- CM: Configuration Management
- CCB: Change Control Board
- CR: Change Request
- CI: Continuous Integration
- Baseline: 배포 가능한 상태로 고정된 형상(태그 + 아티팩트)

## 4. 시스템 개요

- 형상관리 도구: Git (권장: GitHub/GitLab/사내 Git 서버)
- CI/CD: GitHub Actions 또는 Azure Pipelines(프로젝트 정책에 따름)
- 아티팩트 저장소: GitHub Releases 또는 사내 아카이브(릴리스 바이너리 보관)
- 백업/미러: 정기적인 원격 미러 및 `git bundle` 백업
- 접근 제어: 조직의 IDP(예: Azure AD, SSO)를 연동하여 팀/그룹 기반 권한 관리
- 감사 로그: Git 호스팅 서비스의 감사 로그 및 CI 로그 보관

## 5. 저장소 구성

권장 레이아웃(soorak-sam2 리포지토리 정책):

- / (루트)
  - src/ : 소스 코드
  - include/ : 공용 헤더·타입 정의
  - docs/ : 사용 설명서·프로세스 문서
  - wp-templates/ : 사내 워크패키지 템플릿
  - tests/ : 단위/통합 테스트
  - tools/ : 운영 및 빌드 도구
  - build/ : 빌드 산출물(저장소에는 바이너리 제외 권장)
  - releases/ : 릴리스 패키지(태그와 연계된 아카이브)
  - .github/ : 워크플로우 및 설정

디렉터리 규칙:

- 문서와 소스는 분리되어야 하며, 빌드 산출물은 가능한 한 별도 아티팩트 스토어에 보관한다.
- 릴리스 시 관련 메타(릴리스 노트, 체크섬)는 `releases/`에 함께 저장한다.

## 6. 브랜치·베이스라인 전략

- 메인 브랜치

  - `main` : 배포 가능한 최신 베이스라인(프로텍트, 직접 푸시 금지)
  - `develop` : 통합 개발 브랜치(옵션)

- 작업 브랜치

  - `feature/<ticket>-short-desc`
  - `bugfix/<ticket>-short-desc` 또는 `hotfix/<version>`
  - `release/<version>` : 릴리스 준비용

- 베이스라인(릴리스) 생성
  1. `release/<version>` 생성 → CI 통과 → 품질/검토 → `main`으로 병합
  2. 병합 후 주석된(annotated) 태그 생성: `v1.2.0` (태그 메시지에 릴리스 노트와 CR/티켓 참조 포함)
  3. 릴리스 아티팩트(패키지, 체크섬)를 `releases/` 및 아티팩트 스토어에 업로드

## 7. 접근 권한 및 역할

역할과 권한 매핑(권장):

- Owners / Admins

  - 권한: 리포지토리 설정, 브랜치 보호 규칙 관리, 팀/멤버 권한 변경
  - 대상: 조직 관리자, 프로젝트 리더

- Maintainers

  - 권한: PR 승인·병합(단, 보호 브랜치 규칙 준수), 릴리스 브랜치 생성

- Developers

  - 권한: 기능 브랜치 생성·푸시, PR 생성

- Reviewers / QA

  - 권한: PR 리뷰 및 승인(테스트·검증 기준 충족 여부 확인)

- CI / Service accounts
  - 권한: CI 상태 업데이트, 릴리스 아티팩트 업로드(필요 최소 권한)

권한 설정 권고사항(예: GitHub):

- `main` 브랜치 보호: PR 승인(최소 1~2명), 모든 상태 체크 통과, force-push 금지
- 관리자도 보호 규칙을 우회하지 못하도록 설정(필요 시 CCB 승인 절차 별도 적용)
- 민감 작업(릴리스 태깅, 베이스라인 변경)은 별도 팀/CCB 승인 필요

## 8. 변경 이력(커밋/PR) 관리 방법

원칙:

- 모든 변경은 이슈/CR과 연계한다. 커밋 메시지·PR 제목/본문에 CR/이슈 ID를 반드시 명기한다.
- 커밋 메시지 포맷: `CR-<id>: <type>(<scope>): <subject>` 또는 Conventional Commits 권장
  - 예: `CR-123: feat(calculator): add negative-number support`

PR 템플릿 체크리스트(예):

- 관련 이슈/CR 번호 기입
- 변경 요약 및 영향 범위(추적성 링크 포함)
- 단위 테스트/통합 테스트 통과 여부
- 코드 리뷰어 지정 및 승인
- 문서(문서화 필요 시) 업데이트 여부

이력 기록과 산출물:

- 모든 릴리스는 태그와 릴리스 노트로 문서화한다.
- 자동으로 CHANGELOG.md를 생성하도록 CI를 구성할 수 있다(커밋 메시지 기반).
- 태그는 서명(GPG) 또는 CI 서비스 계정으로 생성하여 무결성 확보 권장.

## 9. 변경 통제(승인) 절차

- 주요 변경(주요 아키텍처 변경, 프로덕션 릴리스, 보안 패치)은 CR을 생성하고 CCB(또는 지정된 승인자)의 승인을 받아야 한다.
- CCB 승인 후 릴리스 브랜치 생성 및 테스트, 승인 완료 시 `main` 병합과 태깅 실행.
- 변경 이력(CR 번호, 승인자, 시각)은 변경 통제 기록(WP-13-21)에 기록하고 추적한다.

## 10. 백업 및 보관 정책

- 주기적 원격 미러: 하루 1회(또는 프로젝트 규정에 따라) 다른 호스트로 `git push --mirror` 수행
- 주기적 번들: 주간 `git bundle create repo-YYYYMMDD.bundle --all` 생성 및 안전한 오브젝트 스토어에 보관
- 릴리스 아티팩트는 체크섬(sha256)과 함께 장기 보존 정책에 따라 보관
- 보존기간과 삭제 정책은 프로젝트 운영정책에 따라 문서화

## 11. 감사·추적성

- 요구사항·디자인·테스트 케이스 간 양방향 추적성 필수: 커밋/PR은 요구사항 ID(SWE-REQ-xxxx) 또는 WP 항목을 참조
- CI 파이프라인에서 매 빌드 시 추적성 레포트(커밋→요구사항→테스트)를 생성하여 `docs/traceability/` 등에 저장
- 감사 로그(누가, 언제, 어떤 권한으로 무엇을 변경했는지)는 호스팅 서비스의 감사 로그 및 CI 로그에서 확보

## 12. 도구화 및 자동화 권고

- PR 템플릿, 커밋 규칙 검사(자동 라벨링), 자동 CHANGELOG 생성 도구 사용
- 보호 브랜치에 대해 필수 상태 체크(단위테스트, 정적분석, coverage) 설정
- 릴리스 파이프라인에서 아티팩트 서명·업로드·체크섬 생성 자동화

## 13. 권한 변경 및 예외 처리 절차

- 일시적 권한 상승은 문서화된 요청(CR)으로만 승인하며 만료 시간을 명시한다.
- 예외 승인 기록은 변경 통제 기록에 남기고, 만료 시 자동으로 권한 회수.

## 14. 감사 준비 및 정기 점검

- 분기별 형상관리 감사를 수행하여 베이스라인 무결성, 접근 권한, 변경 이력의 완전성 검증
- 점검 결과는 SUP-8 관련 기록으로 보관하고, 필요 시 시정조치 등록(SUP-1 연계)

## 부록 A — 예시 커맨드/워크플로우

1. 기능 개발 브랜치 생성 및 PR

```bash
git checkout -b feature/CR-123-add-neg
git add .
git commit -m "CR-123: feat(calculator): add negative number support"
git push origin feature/CR-123-add-neg
# Git hosting에서 PR 생성 → 리뷰 → CI 통과 → 병합
```

2. 릴리스 베이스라인 생성(예시)

```bash
# release 브랜치로 RC 생성
git checkout -b release/1.2.0 develop
# 릴리스 준비(테스트·문서·검토)
git push origin release/1.2.0
# 승인이 완료되면
git checkout main
git merge --no-ff release/1.2.0
git tag -a v1.2.0 -m "Baseline v1.2.0 - CRs: CR-120, CR-123"
git push origin main --tags
```

3. 백업 예시

```bash
git bundle create /backup/soorak-sam2-$(date +%Y%m%d).bundle --all
# 또는 원격 미러로 주기적 push
git push --mirror git@backup.example.com:soorak-sam2-mirror.git
```
