## 1. 목적

## 3. 범위

- In-Scope: 소스 레벨의 SW 아이템 통합(단위 통합 → 모듈 통합 → 시스템 통합), 통합 시험(인터페이스·기능·회귀), 통합 빌드 산출물.
- Out-of-Scope: 하드웨어 통합(ECU), 외부 시스템 연동(타 시스템과의 실제 네트워크 통합, 별도 계약 대상).

가정: 각 단위(SWE-UNIT)별 단위 테스트는 사전에 구현·통과되어 있다(또는 통과 가능한 상태로 수정한다).

## 4. SW 아이템 식별

아래는 본 프로젝트의 통합 대상 SW 아이템(예시)이다.

- `types` — 공유 타입 정의 (src/types.h) — SWE-UNIT-0004
- `Calculator` — 사칙연산 모듈 (src/Calculator.h/.cpp) — SWE-UNIT-0001
- `InputParser` — 입력 파서 (src/InputParser.h/.cpp) — SWE-UNIT-0002
- `AppController` — REPL/앱 제어기 (src/AppController.h/.cpp) — SWE-UNIT-0003
- `main` — 프로그램 진입점 (src/main.cpp)
- `Integration Test Harness` — 통합 테스트 바이너리 및 스크립트 (tests/integration/)

각 아이템은 빌드 단위(라이브러리/오브젝트)로 정의되며, 통합 시점에 필요한 인터페이스 명세(함수 시그니처, 데이터 구조)를 참조한다.

## 5. 통합 전략

원칙: 점진적(incremental)·하향(top-down) 또는 상향(bottom-up)의 혼합을 사용하되, 재현성 있는 자동화 빌드와 자동화 테스트를 우선 적용한다.

- 통합 방식: Bottom-up 우선 — 핵심 계산 로직(`Calculator`) 검증 후 파서(`InputParser`)와 통합, 그 다음 `AppController` 및 `main`으로 확장.
- 자동화: CMake 기반 빌드와 `ctest`/Google Test 자동화, CI(옵션)에서 빌드·테스트 파이프라인 실행.
- 빌드 무결성: 각 통합 단계마다 빌드 리스트(build list)를 생성하고, 해당 빌드 산출물(정적 라이브러리, 실행파일)을 아카이브한다.

## 6. 통합 순서 (권장)

1. 준비 단계 — 단위시험 통과

   - 진입 기준: 각 유닛의 단위 테스트가 통과하거나 동작상의 심각한 결함이 없음
   - 산출물: 단위 테스트 결과 리포트

2. 단계 A: `Calculator` 단독 통합 확인

   - 목적: 연산 정확성 및 경계값(overflow, divide-by-zero) 확인
   - 진입 기준: Calculator 단위 테스트 통과
   - 종료 기준: 기능 ITC(통합테스트) 통과, 중요 결함 없음

3. 단계 B: `Calculator` + `InputParser` 통합

   - 목적: 문자열 입력 → 파싱 → 계산 흐름의 인터페이스 검증
   - 진입 기준: 단계 A 종료, InputParser 단위 테스트 통과
   - 시험 항목: 정상 입력, 잘못된 입력, 경계값, 예외(QUIT 등)
   - 종료 기준: ITC 통과, 인터페이스 결함 해소

4. 단계 C: `AppController` 통합 (REPL 동작)

   - 목적: 사용자 시나리오(프롬프트 → 입력 → 결과 → 반복) 검증
   - 진입 기준: 단계 B 종료
   - 종료 기준: 시나리오 기반 통합 테스트 통과

5. 단계 D: `main` 포함 시스템 통합
   - 목적: 전체 정합성, 시작/종료 시퀀스, 에러 핸들링 검증
   - 진입 기준: 단계 C 종료
   - 종료 기준: 전체 통합 테스트 통과, 고(高)심각도 미해결 결함 없음

각 단계에서 통합 빌드 스냅샷을 보관하고, 테스트 로그와 트레이스파일을 추적한다.

## 7. 시험 전략 및 방법 (SWE-5 BP2 준수)

목적: 본 절은 소프트웨어 통합 시험의 설계·선택·실행 방법을 상세화하여 SWE-5 BP2 요구사항을 충족하는 통합 시험 전략을 제시한다. 모든 통합 ITC(Integration Test Case)는 요구사항 및 단위 설계와 추적 가능한 매핑을 가져야 한다.

- 시험 레벨 및 목적:

  - 단위(Unit): 유닛별 기능·경계값 확인(사전 전제)
  - 통합(Integration): 모듈 간 인터페이스·상호작용·데이터 흐름 검증
  - 시스템(System): 전체 시나리오·시퀀스·에러 경로 검증
  - 회귀(Regression): 변경 영향 검증 및 기존 기능 보호

- 통합 테스트 유형:

  - 인터페이스/계약 검증: 함수 시그니처, 에러 코드, 데이터 포맷
  - 상호작용 시나리오: 입력 → 파싱 → 계산 → 출력의 연속 흐름
  - 비정상/예외 경로: 잘못된 입력, 경계값, 예외조건(QUIT 등)
  - 성능/응답성(필요시): REPL 지연 및 처리시간 측정

- 시험 설계 원칙:

  - Traceability: 각 ITC는 요구사항 ID 및 관련 `SWE-UNIT` 설계 ID와 매핑
  - 재현성: 자동화 스크립트로 동일 환경에서 반복 실행 가능
  - 최소권한·격리: 외부 의존은 모킹/스텁으로 격리
  - 가독성: 입력·예상결과·사전조건·후조건을 명확히 기술

- ITC 형식(권장 템플릿):

  - ITC ID: `ITC-SWE5-0001`
  - 제목: 간결한 한줄 설명
  - 관련 요구사항: `SWE-REQ-xxxx`
  - 관련 단위설계: `SWE-UNIT-xxxx`
  - 전제조건: 빌드·환경·데이터
  - 입력값/시나리오
  - 기대값(검증 기준)
  - 실행 절차(자동/수동)
  - 로그·증거(파일·스크린샷 경로)

- 테스트 우선순위 및 선택 기준:

  - 우선순위 기준: 안전성·정확성 > 인터페이스 안정성 > 보조 기능
  - 선택 기준: 최근 변경 영향(커밋 범위), 주요 요구사항 커버리지, 결함 히스토리

- 자동화 전략:

  - 프레임워크: `CMake` + `Google Test` + `CTest`를 기본으로 사용
  - 통합 테스트는 `tests/integration/` 하위의 Google Test 기반 테스트로 구현
  - 테스트는 `ctest`로 실행 가능해야 하며, CI 파이프라인에서 자동 호출
  - 테스트 실행 스크립트는 재현 가능한 환경 구성을 포함(환경 변수, 입력 파일 경로)

- 테스트 환경 및 시뮬레이션:

  - 권장 환경: Debug 빌드(CMake 3.16+, GCC/G++ 또는 MSVC), Google Test v1.14.0
  - 외부 의존(파일 입출력 등)은 테스트용 더미/모의 구현으로 대체
  - Windows 환경 대응: MSVC 빌드/테스트 스크립트 제공

- 커버리지 및 품질 목표:

  - 통합 주요 모듈: 라인/분기 커버리지 권장 ≥ 80% (중요 기능은 더 높게 설정)
  - 커버리지 수집: `lcov`/`gcovr` 로 리포트 생성 및 보관

- 시험 산출물:

  - ITC 명세서, 테스트 스크립트/바이너리, 실행 로그, 커버리지 리포트, 결함 리포트

- 지표 및 보고:
  - 매 실행: 총 ITC 수, PASS/FAIL, 커버리지(라인/분기), 실행시간
  - 회귀 결과: 변경별 영향 요약 및 실패 테스트 목록

위 규칙을 통하여 통합 단계별로 일관된 시험 설계·실행·추적성을 확보한다.

## 8. 완료/종료 기준

- 각 통합 단계별 종료 기준 예시:
  - 모든 해당 ITC가 `PASS`
  - 우선순위 높은(Severity 1) 결함 0건
  - 누적 미해결 결함 수가 프로젝트 규정 내 허용 범위 이내
  - 통합 빌드 및 테스트 로그가 저장소에 등록

최종 시스템 통합 종료 기준:

- 전체 통합 테스트 통과(설정된 커버리지 목표 충족 — 권장: 라인 커버리지 ≥ 80%)
- 모든 Critical/High 결함 해결 또는 승인된 예외 처리

## 9. 시험 데이터 및 환경 요구사항

- 테스트 데이터: 정상 수식(정수·실수), 음수, 0, 큰 수, 잘못된 형식(문자열), 빈 입력, QUIT 명령 등
- 테스트 환경(권장):
  - 빌드 도구: CMake 3.16+
  - 컴파일러: GCC/G++ (또는 MSVC for Windows 환경)
  - 테스트 프레임워크: Google Test (FetchContent로 관리)
  - 테스트 실행: `ctest` / 개별 Google Test 바이너리
  - 커버리지 도구: gcov/lcov 또는 gcovr

실행(예):

```bash
# Debug 빌드
cmake -B build -DCMAKE_BUILD_TYPE=Debug
cmake --build build

# 테스트 실행
ctest --test-dir build -V
```

## 10. 회귀 시험 전략 (상세)

목적: 코드 변경이 기존 동작을 손상하지 않도록 회귀 시험을 체계화하고 자동화한다.

- 회귀 유형 및 실행 빈도:

  - Smoke (Tier 0): 빌드·핵심 기능 확인 — 모든 커밋에서 실행(빠른 피드백)
  - Impacted-area Regression (Tier 1): 변경된 파일/모듈에 영향받는 ITC만 실행 — PR 빌드에서 실행
  - Integration Regression (Tier 2): 통합 시 영향도가 큰 흐름(예: Calculator+Parser 시나리오) — Nightly 또는 PR 병행
  - Full Regression (Tier 3): 전체 ITC + 커버리지 수집 — 릴리스 후보 및 정기(예: nightly) 실행

- 회귀 트리거 정책:

  - Commit push: Tier 0(및 가능한 경우 Tier1) 자동 실행
  - Pull Request: Tier 0 + Tier 1 자동 실행(병렬화 권장), PR 승인 전 모든 Tier1 통과 권장
  - Merge to main/release branch: Tier 0~2 자동 실행, Release 후보에 대해 Tier 3 실행
  - 수동 트리거: QA 요청·핫픽스/릴리스 전 전체 회귀

- 테스트 선택(Impacted-area) 전략:

  - 변경영역 매핑: 변경된 소스 파일 → 소속 모듈 → 관련 ITC 매핑표 사용
  - 빠르게 선택된 ITC만 실행해 PR 피드백 시간을 단축

- 안정성 및 flaky 테스트 관리:

  - 재시도 정책: 동일 테스트 2회 재실행(시간 내), 2회 실패 시 결함 등록
  - 격리(Quarantine): 반복적으로 불안정한 테스트는 격리하여 별도 조사

- 시간·자원 최적화:

  - 병렬 실행: `ctest --parallel` 또는 CI 병렬 잡 활용
  - Tier 기준으로 실행 우선순위 설정(빠른 피드백 우선)

- 회귀 증거 및 추적:

  - 각 회귀 실행은 커밋 SHA, 빌드 ID, 수행된 ITC 목록과 결과, 커버리지 리포트를 포함
  - 실패 ITC는 결함 보고서(SUP-9)로 등록하고 원인·우선순위를 지정

- 베이스라인 관리:
  - 성공 기준 베이스라인을 정의하고 주요 릴리스별 베이스라인 스냅샷을 보관

이 전략을 통해 변경에 따른 위험을 빠르게 발견·완화하고, 릴리스 품질을 보장한다.
