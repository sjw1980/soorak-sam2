# WP.08-52 — SW 단위 검증 전략 및 시험 계획서

**버전:** v1.0 / 2026-04-19

**상태:** Draft

**관련 BP:** SWE.4.BP1 (단위 검증 전략)

## 1. 목적

이 문서는 소프트웨어 단위(Unit) 레벨의 검증 전략 및 시험 계획을 정의한다. 본 계획서는 단위 테스트의 범위, 시험 기법·도구, 회귀(Regression) 전략, 검증 방법, 커버리지 목표 및 측정 방법을 규정하여 SWE-4 요구사항을 충족하는 것을 목적으로 한다.

## 2. 범위

- 적용 대상: `src/`에 해당하는 단위 코드를 포함한 모든 SW 유닛
- 포함: 자동화된 단위 테스트, 정적 분석, 커버리지 수집 및 레포트
- 제외: 시스템 통합 테스트(SWE-5), 적격성 테스트(SWE-6)

## 3. 가정 및 제약

- 빌드 도구: CMake, 컴파일러: GCC/G++(프로젝트 표준)에 기반한다.
- 단위 테스트 프레임워크: Google Test 사용을 권장한다.
- 커버리지 측정: Debug 빌드에 gcov/lcov 또는 gcovr 사용.
- CI 환경에서는 병렬 빌드 및 테스트가 가능하다고 가정한다.

## 4. 책임

- 개발자: 단위 테스트 케이스 작성·수정, 로컬 실행 및 초기 실패 해결
- 테스트 리드: 시험 계획 유지·검토, 회귀 전략 운영, 커버리지 목표 검증
- 빌드/CI 엔지니어: 자동화 파이프라인 구성, 리포트 보관

## 5. 시험 전략 (방법/기법/도구)

- 정적 검증
  - 도구: `clang-tidy`, `cppcheck`, 정적 분석 도구. MISRA/코딩 규칙 체크는 별도 정적검증 절차로 연계.
- 동적(단위) 검증
  - 프레임워크: Google Test
  - 기법: 화이트박스(구조 기반) 테스트 + 블랙박스(요구사항 기반) 테스트의 조합
  - Mocking: 외부 의존성/하드웨어 의존 코드는 Mock 또는 Stubs 사용
- 구조 기반 검증
  - 경계값, 예외 처리, 에러 경로, 분기 경로 집중 검증
- 자동화·CI 통합
  - 모든 PR에 대해 자동 단위 테스트(빠른 하위집합) 실행
  - Merge/PR 승인 전, main 브랜치로의 병합은 전체 단위 테스트 통과 및 커버리지 가드 충족 필요

## 6. 회귀 시험 전략

1. 실행 빈도 및 트리거

- Commit/PR 레벨: 변경 범위 기반의 빠른 단위 테스트 서브셋(핵심·영향 모듈) 실행 — 빠른 피드백
- Merge/Pre-merge: 전체 단위 테스트 스위트 실행 (Gating)
- Nightly: 전체 스위트 + 커버리지 수집·리포트
- Weekly/Release: 확장 회귀(타깃/하드웨어 포함) 실행

2. 케이스 선정

- 변경 영향 기반 선택: 변경된 파일/함수에 매핑된 테스트 우선 실행
- 중요/안전 관련 모듈: 변경 여부와 무관하게 전체 실행(Full regression)

3. 실패·불안정(Triage)

- 실패 시: 자동으로 이슈(또는 CI 실패 로그)를 생성하고 담당자 할당
- 플래키(불안정) 테스트: 식별 후 잠정 격리(Quarantine)하여 로그·이슈 작성, 원인 분석·수정 후 재통합

4. 출입 규칙(Gating)

- Merge 금지 조건: 필수 단위테스트 실패 또는 커버리지 최소 기준 미달

## 7. 검증 방법 (구체적 절차)

1. 요구사항-테스트 매핑

- 각 단위 요구사항(또는 상세설계 항목)은 최소 하나 이상의 단위 테스트로 검증되어야 한다.
- 추적성 레코드는 [docs/wp-templates/WP-13-22.md](docs/wp-templates/WP-13-22.md) 형식으로 유지.

2. 정적 분석

- CI 단계에서 정적분석을 실행하고, 새로운 심각도 높은 경고는 빌드 실패 기준으로 검토.

3. 동적 테스트 실행

- 로컬 개발: 개발자가 로컬에서 테스트 및 커버리지 실행
- CI: 자동화된 테스트 실행 및 아티팩트(테스트 로그·커버리지 리포트) 업로드

4. 검토·인증

- 테스트 설계는 코드 리뷰에서 점검 (검증 커버리지, 경계조건, 예외경로 등)

## 8. 커버리지 목표 및 측정

1. 목표(프로젝트 기준)

- 문장(Statement) 커버리지: 목표 90% (최소 허용 80%)
- 분기(Branch) 커버리지: 목표 85% (최소 허용 75%)
- 함수(Function) 커버리지: 목표 95%
- MC/DC: 안전·안전대상(Safety-related) 모듈에 한해 적용 (적용 시 목표 75% 이상)

2. 요구사항 커버리지

- 단위 수준으로 할당된 모든 요구사항은 테스트 케이스로 매핑되어야 하며, 매핑율 100% 목표

3. 측정 방법

- 빌드: Debug + 커버리지 플래그 (예: `-fprofile-arcs -ftest-coverage`)
- 도구: `lcov` + `genhtml` 또는 `gcovr`로 HTML 리포트 생성
- 제외: 외부 라이브러리, 테스트 프레임워크 소스, 시뮬레이터/세 번째 파티 코드

4. 커버리지 리포트 생성 예시

```bash
# Debug 빌드, 커버리지 플래그는 CMake 설정에서 활성화
cmake -B build -DCMAKE_BUILD_TYPE=Debug
cmake --build build
ctest --test-dir build -V

# lcov 예시
lcov --capture --directory build --output-file coverage.info \
     --exclude '/usr/*' --exclude '*/gtest/*' --exclude '*/third_party/*'
lcov --remove coverage.info '*/tests/*' --output-file coverage.filtered.info
genhtml coverage.filtered.info --output-directory build/coverage_report

# gcovr 대체 예시
gcovr --root . --exclude '.*third_party.*' --html --html-details -o build/coverage_report/index.html
```

## 9. 시험 환경 및 CI 통합

- 로컬 개발: 위의 `cmake` / `ctest` 명령으로 실행
- CI(예: GitHub Actions/Azure Pipelines):
  - Commit/PR 단계: 빠른 서브셋 실행
  - Merge/Protected branch: 전체 스위트 + 커버리지 측정
  - Nightly: 전체 스위트 + coverage HTML 아티팩트 저장

## 10. 시험 케이스 관리·식별

- TC ID 형식: `SWE-TC-YYYY-XXXX` (프로젝트 규칙에 따라 부여)
- 각 TC는 목적, 전제조건, 절차, 예상결과, 우선순위를 포함
- 테스트 케이스 템플릿 참조: [docs/wp-templates/WP-08-50.md](docs/wp-templates/WP-08-50.md)

## 11. 결함 처리 및 추적

- 실패한 테스트는 자동으로 이슈 또는 결함 레포트에 링크되어야 하며, SUP-9 프로세스(문제관리)를 통해 추적
- 결함 우선순위는 심각도·재현성·영향 범위에 따라 결정

## 12. 수용 기준

- 모든 단위 테스트 통과
- 커버리지 목표(또는 최소 허용치) 충족
- 주요 결함(High/Critical)이 해결되어 검증 완료

## 13. 추적성

- 단위 테스트 ↔ 상세설계/요구사항 간 양방향 추적성 확보 (추적성 문서 참조: [docs/wp-templates/WP-13-22.md](docs/wp-templates/WP-13-22.md))
